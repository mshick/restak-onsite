# CLAUDE.md

This repo is for preparing for the **Restak virtual onsite** — a 3-hour working session that includes a ~75-minute build of a tool helping insurance brokerages reconcile documents against their system of record.

## Read this first

`docs/onsite-prep.md` — the full problem framing, likely scenarios, discrepancy taxonomy, and v1 sketch. Read it before suggesting anything; it's the working brief from a prior planning session and the assumptions there should be treated as decided unless I explicitly reopen them.

## Stack preferences for the prototype

Personal **Next.js + Supabase + Tailwind + ShadCN** starter. Don't fight it.

- Next.js (app router) for flexibility — route handlers and server actions available if needed, no penalty if the build stays client-side
- Supabase from the start — one denormalized table, no auth, single mock user. Persistence is worth the small cost (survives HMR reloads, makes the demo land).
- Tailwind + ShadCN for styling — perception matters and the starter saves setup time
- TanStack Query for data flow against the Supabase wrapper
- pnpm

Use ShadCN component defaults; don't burn time on variant tuning. The design lives in information hierarchy, not in chrome.

## How I want to work

- Push back on choices that don't fit the 75-minute window
- When in doubt about scope, ask before building
- Default to minimal dependencies; flag it if you want to add one
- Narrate non-obvious decisions in commit messages or inline comments so I can replay reasoning during the onsite

## Out of scope for this repo

- Document extraction (the onsite will provide already-extracted JSON)
- Auth, real persistence, deployment
- Anything that takes more than ~10 minutes of setup before the first useful UI shows up

## Repo layout

The scaffold is a stripped-down clone of `~/Code/mshick/dawn` with auth, chat,
document ingest, and Inngest removed. What's left:

```
src/
  app/
    page.tsx               # Reconciliation queue (server component, reads from supabase)
    items/[id]/            # Detail view: side-by-side fields + discrepancies
    api/reconcile/         # POST /api/reconcile — runs the pipeline
  components/ui/           # ShadCN (button only — add more via `pnpm dlx shadcn@latest add`)
  lib/
    reconcile/             # ← THE LLM SEAM. Read its README first.
      rules.ts             #   Deterministic pre-pass (string norm, tolerance, sets)
      llm.ts               #   OpenAI call (@ai-sdk/openai, gpt-5.4-mini)
      prompts.ts           #   System prompt + per-call user prompt
      types.ts             #   5-tier discrepancy taxonomy
      index.ts             #   reconcile(sor, ext) — public surface
    supabase/{server,client,admin}.ts   # Supabase JS clients (user, browser, service-role)

```

## The data model

One denormalized table — `reconciliation_items`. The two payloads being
compared (`system_of_record`, `extracted`) are both `jsonb`, so the onsite can
hand us any field shape. Computed comparison output goes back into
`discrepancies jsonb` on the same row. No joins, no other tables.

## The LLM seam

`src/lib/reconcile/` is where Claude lives. The pipeline is shaped for the
**semi-structured** input the onsite will hand us (per-field extraction
envelopes with `raw_text` + `confidence`, plus free-text
`unparsed_sections`). In that world the LLM is the **primary comparator**;
rules are only a pre-filter.

1. `rules.ts` — pre-filter. Emits only `cosmetic` (string normalization
   match) and `auto_resolved` (equivalent date format, sub-tolerance numeric
   drift). Anything else, including any field with extraction confidence
   below 0.7, goes to the LLM.
2. `llm.ts` — OpenAI (`gpt-5.4-mini`) does the judgment work. Sees the
   residue fields **and** all `unparsed_sections`. Emits two kinds of
   finding:
   - `kind: 'field'` — one per residue field. If the LLM judges them
     equivalent it still emits a `cosmetic` finding (reasoning lands in the
     audit log).
   - `kind: 'narrative'` — material changes the model spotted in free-text
     sections that did not surface as a structured field. Quotes a verbatim
     excerpt. This is what catches "exclusion silently added in the cover
     letter" — invisible to a pure-diff tool.

Uses `generateText` with `Output.object({ schema })` and a zod
discriminated-union schema, so the LLM's output is structurally validated.
No free-form JSON parsing.

Fall-back: with no `OPENAI_API_KEY` the LLM stub flags every residue
field as `tier: 'ambiguous'`. Narrative findings cannot be faked — they
only appear when OpenAI is reachable.

If the scenario calls for richer LLM use (entity resolution across records,
multi-doc comparison, etc.), the change happens in `src/lib/reconcile/`
only — callers don't move.

## Run

```
pnpm install
cp .env.example .env.local        # paste keys from `pnpm db:start`
pnpm db:start                     # local Supabase (Docker)
pnpm db:reset                     # apply migration + seed
pnpm dev                          # http://localhost:3000
```
